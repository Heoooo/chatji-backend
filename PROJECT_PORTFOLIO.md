# Chatji(찾지) - 실시간 맞춤형 핫딜 스나이핑 서비스

## 1. 프로젝트 개요
- **프로젝트명:** Chatji (찾지)
- **한 줄 소개:** "당신이 잠든 사이에도 최저가를 찾습니다." - 사용자가 설정한 조건(키워드, 카테고리, 목표가)에 맞는 핫딜을 실시간으로 감시하고 즉시 알려주는 지능형 쇼핑 비서 서비스입니다.
- **개발 배경:** 기존 쇼핑 검색 서비스(네이버 쇼핑 등)는 사용자가 직접 검색해야 하는 '수동적' 구조이며, 실제 가격 변동폭이 작아 '진짜 할인'을 체감하기 어렵다는 한계가 있었습니다. 이를 해결하기 위해 여러 커뮤니티의 핫딜 정보를 실시간으로 수집하고, 개인화된 조건에 맞춰 '능동적으로' 알려주는 스나이핑 서비스로 기획을 고도화(Pivot)했습니다.

## 2. 주요 기능
- **🎣 지능형 알림(낚싯대) 설정:** 키워드(예: 맥북), 카테고리(예: 가전 > 노트북), 목표가(예: 50,000원 이하) 등 다양한 조건으로 알림을 설정할 수 있는 대시보드 제공.
- **🔥 실시간 핫딜 매칭 엔진:** 뽐뿌, 루리웹 등 대형 커뮤니티의 핫딜 게시판을 크롤링하여, 수집 즉시 사용자의 알림 조건과 대조 및 매칭.
- **🔔 SSE 기반 실시간 푸시 알림:** Server-Sent Events(SSE)를 활용하여 매칭된 핫딜 정보를 클라이언트(브라우저)에 지연 없이 푸시 알림으로 전달.
- **📊 핫딜 스코어링 및 시세 분석:** 수집된 핫딜 가격을 네이버 쇼핑 최저가와 비교하여 할인율(Discount Rate)을 계산하고, 자체적인 '핫딜 점수(Score)'를 부여하여 신뢰도 제공.
- **🎁 사용자 행동 기반 맞춤 추천:** 사용자의 클릭(Click) 및 관심(Wish) 데이터를 분석하여 선호 카테고리를 도출하고, 해당 분야의 고득점 핫딜을 우선 추천.

## 3. 기술 스택
- **Backend:** Java 17+, Spring Boot, Spring Data JPA
- **Database / Cache:** MySQL, Redis (검색 결과 캐싱)
- **Frontend:** React, Vite, Recharts (시세 차트 시각화), Vanilla CSS (Glassmorphism UI)
- **Infra / Library:** Resilience4j (Circuit Breaker), Jsoup (Web Crawling), SSE (Server-Sent Events)

## 4. 💡 기술적 도전 및 트러블 슈팅 (자소서 핵심 어필 포인트)

### ① 기획의 피벗(Pivot)과 데이터 가치 재발견
- **문제:** 초기에는 네이버 쇼핑 API를 단순 호출하는 구조였으나, 기존 대형 서비스와의 차별성이 부족하다는 피드백을 수용.
- **해결:** 데이터의 성격을 '정적인 카탈로그'에서 '동적인 커뮤니티 핫딜'로 전환. 네이버 쇼핑은 '기준 가격'을 확인하는 용도로만 사용하고, 메인 가치를 **"수많은 핫딜 중 내게 필요한 것만 골라내는 필터링"**으로 재정의하여 서비스의 정체성을 확립함.

### ② SSE를 활용한 실시간 알림 엔진 최적화
- **문제:** 폴링(Polling) 방식은 서버 부하가 심하고 실시간성이 떨어짐.
- **해결:** **SSE(Server-Sent Events)**를 도입하여 서버에서 클라이언트로의 단방향 실시간 데이터 스트리밍 파이프라인을 구축함. 크롤러가 새로운 핫딜을 DB에 저장하는 트랜잭션 내에서 매칭 로직(`AlertService`)을 비동기적으로 호출하여, 조건이 일치하는 유저의 SSE 커넥션으로 즉시 알림을 발송하도록 구현.

### ③ 외부 API 의존성 극복과 가용성 확보 (Resilience4j)
- **문제:** 네이버 쇼핑 API의 Rate Limit이나 일시적 장애가 전체 서비스의 장애로 이어지는 문제 발생 (Cascading Failure).
- **해결:** **Resilience4j의 Circuit Breaker**를 도입. 외부 API 호출 실패율이 임계치를 넘으면 서킷을 오픈(Open)하여 빠른 실패(Fail-fast)를 유도하고, Fallback 로직을 통해 빈 리스트나 캐시된 데이터를 반환하도록 처리하여 시스템의 전체 가용성(Graceful Degradation)을 유지함.

### ④ 비정형 데이터 정규화 및 카테고리 매칭 매커니즘
- **문제:** 커뮤니티마다 작성하는 핫딜 상품명 형식이 다르고(예: 괄호, 특가 표기 등), 자체적인 카테고리 분류가 명확하지 않음.
- **해결:** 정규 표현식을 활용한 텍스트 전처리 로직을 구현하여 불필요한 수식어를 제거하고 순수 상품명과 가격을 추출함. 이후 해당 키워드로 네이버 API를 1차 검색하여 정확한 대분류(`categoryLarge`)와 소분류(`categorySmall`) 데이터를 획득, 이를 기반으로 유저의 알림 설정과 정확하게 매칭시키는 정규화 파이프라인을 구축함.

### ⑤ 알림 매칭 엔진 최적화 및 성능 개선 (O(N) → O(Log N))
- **문제**: 서비스 확장 시 알림 설정 유저가 늘어남에 따라, 수집된 핫딜을 모든 유저와 대조하는 루프(Loop) 방식이 서버 응답 지연의 주범이 됨 (1만 건 기준 742ms 소요).
- **해결**: 
    - Java 애플리케이션 레벨의 비교 연산을 **최적화된 DB 쿼리**로 이관.
    - `keyword`, `category`, `active` 컬럼에 **복합 인덱스(Composite Index)**를 구축하여 검색 공간을 최소화함.
- **성과**: 
    - 매칭 소요 시간: **742.34ms → 12.18ms** (약 **98% 성능 향상**)
    - 대규모 알림 설정 환경에서도 일정한 성능을 유지할 수 있는 확장성(Scalability) 확보.

## 5. 🤖 생성형 AI(바이브 코딩) 활용 경험
...
- AI를 단순히 코드를 생성하는 도구가 아닌, **'기술적 의사결정을 돕는 파트너'**로 활용했습니다.
- 프로젝트 초기 차별성 부족 문제를 AI와 토론하며 '실시간 핫딜 스나이퍼'라는 구체적인 기획으로 피벗(Pivot)할 수 있었습니다.
- 대규모 리팩토링(카테고리 체계 세분화 등) 과정에서 AI의 전역 컨텍스트 분석을 활용하여 DB 엔티티, 비즈니스 로직, 프론트엔드 UI를 일관성 있게 동시 수정하며 휴먼 에러를 예방하고 개발 주기를 크게 단축시켰습니다.
- SSE, Circuit Breaker와 같은 복잡한 아키텍처 도입 시, AI를 통해 보일러플레이트 코드를 빠르게 구성하고 저는 **분산 환경에서의 정합성 보장**과 **비즈니스 로직 최적화**라는 고차원적인 고민에 집중하여 생산성을 극대화했습니다.
